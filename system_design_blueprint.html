Scope & Requirements
Define what you're building before anything else
Foundation ▶
Why this comes first: Every architectural decision downstream is hostage to scope. A chat app that also needs video calls is a fundamentally different system than one that doesn't. Nail the scope, or you'll redesign twice.
Functional scope
✓
What are the core features you must support at launch?
List them. Be ruthless about cutting non-essential features. Every feature you add is an architectural constraint.
critical
✓
What are the explicit out-of-scope items?
State them out loud. Stops scope creep during design. "Voice calls are out of scope" saves 2 hours of architecture discussion.
✓
Who are the actors in this system?
Users, admins, third-party services, automated bots? Each actor type has different access patterns and trust levels.
Non-functional requirements
✓
What is the acceptable latency for core operations?
p50 / p99 targets. "Fast" is not a spec. "Under 200ms p99 for reads" is. This number decides caching strategy, database choice, CDN need.
critical
✓
What SLA (uptime) are you committing to?
99% = 3.65 days downtime/year. 99.9% = 8.7 hours. 99.99% = 52 minutes. Each nine costs exponentially more to achieve.
critical
✓
Is the system consistency-critical or availability-critical?
CAP theorem forces a choice under partition. Banking = consistency. Social feed = availability. Most apps fall somewhere between.
✓
Are there regulatory or compliance constraints?
GDPR, data residency (India data must stay in India), PCI-DSS for payments, HIPAA for health. These force geographic database placement.
02

Capacity Estimation
Anchor all decisions in real numbers before drawing anything
Numbers First ▶
The rule: Estimate before you design, not after. The numbers reveal which components are load-bearing. A system handling 100 req/sec and one handling 1M req/sec are completely different designs, even for the same feature set.
Traffic volume
✓
How many Daily Active Users (DAUs) at launch, 6 months, 2 years?
Design for 2-year projection. Build for 6-month. Ship for launch. These three numbers are your three design horizons.
critical
✓
What is the peak-to-average traffic ratio?
Most systems have 3–10× spike over average. Your infra must handle peak, not average. A food delivery app spikes 8× at dinner time.
critical
✓
What is the read-to-write ratio?
Twitter/Instagram: 100:1 reads. Chat: 1:1. Logs: 1:100 writes. This single ratio determines your caching depth, replica count, and DB choice.
metric
Storage & data
✓
What is the average payload size per request / record?
Text message = ~100 bytes. Tweet = ~280 bytes. Profile photo = ~500KB. These numbers × daily volume = storage growth rate.
metric
✓
How fast does data grow per day / month?
This tells you when a single DB node will be exhausted and forces the partitioning/sharding conversation. Know this before you pick a database.
metric
✓
Is there a hot data vs cold data pattern?
Last 7 days of messages get 95% of reads. Older data is rarely accessed. Enables tiered storage — hot on SSD, cold on object storage.

Architecture Style Decision
Monolith vs modular monolith vs microservices — answer before picking
One-time decision ▶
The trap: Defaulting to microservices because it sounds impressive. Most systems should start as a modular monolith. Extract services only when you feel actual pain — not imagined future pain.
Team & org questions
✓
How many engineers, and will there be multiple autonomous teams?
1–5 engineers → monolith. 5–20 → modular monolith. 20+ with separate teams → microservices. Conway's Law: your architecture mirrors your org chart.
critical
✓
Do different parts of the system have radically different scaling needs?
If search needs 50× more resources than checkout, you need to scale them independently. That's the strongest technical case for microservices.
✓
Are there any components with special infrastructure needs?
Real-time connections require stateful servers. Media processing needs GPU/high-CPU nodes. ML inference needs specialized hardware. These naturally become separate services.
✓
Are your domain boundaries clear and stable?
Wrong boundaries in microservices = expensive rewrites. If domains are still evolving, a modular monolith lets you refactor without changing network contracts.

Database Selection & Strategy
Pick the right store for each data type, then decide on scaling path
Data layer ▶
The rule: Different data types belong in different stores. Don't put everything in one DB because it's familiar. Then: replicate before you shard. Shard only when replication isn't enough.
Data shape questions
✓
Is your data relational (entities with relationships) or document-like / key-value?
Users + orders + products with foreign keys = relational. User sessions, shopping carts, ephemeral state = key-value (Redis). Event logs, messages = wide-column (Cassandra).
critical
✓
Do you need ACID transactions? Across which operations?
"Deduct inventory AND charge payment" must be atomic. If yes → PostgreSQL. If you can tolerate eventual consistency → Cassandra or DynamoDB open up.
critical
✓
What is the primary query pattern? Point lookup, range scan, or aggregation?
Point lookup (user by ID) → Hash index, Redis. Range scan (orders in date range) → B-tree index, Cassandra range partition. Aggregation → OLAP, ClickHouse.
metric
Scaling path questions
✓
What is the write throughput? Can one DB primary handle it?
A strong Postgres primary handles ~10K–50K writes/sec. Beyond that: sharding. Cassandra: 100K+ writes/sec natively. Know this number before picking your DB.
metric
✓
Can reads tolerate staleness (eventual consistency), or must they always see the latest write?
"Show my profile" reads can be 500ms stale — serve from replica. "Show my bank balance after transfer" cannot — must go to primary. This decides replication type.
✓
Does data need to be co-located geographically? (data residency laws)
EU users' data must stay in EU (GDPR). Indian financial data must stay in India (RBI mandate). This forces geographic sharding or multi-region DB clusters.
✓
What's your partitioning key if you need to shard?
The key must distribute load evenly and match your most common query. user_id for user-centric data. chat_id for messages. Timestamp for time-series. Bad key = hotspots.
critical

⚡
Caching Strategy
Decide what to cache, where, for how long, and how to invalidate
Performance layer ▶
The rule: Cache the result of expensive operations that are read frequently and change infrequently. Never cache what you don't understand — stale cache is often worse than no cache.
What and where to cache
✓
What are your most frequently read, rarely changing data objects?
User profiles, product listings, config data — perfect cache candidates. A user profile read 1000×/hour but updated once/day gives you 999 cache hits per update.
critical
✓
Should the cache be distributed (Redis) or in-process (local memory)?
Multiple app servers → must use Redis (shared state). If you cache locally, Server A's cache and Server B's cache diverge. Single server → local is faster (no network hop).
✓
Is there static media (images, videos, JS/CSS)? Does it need a CDN?
Any content served globally + doesn't change per-user = CDN. CDN serves from the nearest PoP, removes load from your origin, and handles DDoS for free.
Invalidation questions
✓
How stale can cached data be? → Sets your TTL.
Product price can be 30s stale. Your own username shown back to you should be fresh. "How fresh" × "how expensive to recompute" = TTL.
✓
When data changes, can you afford to wait for TTL expiry, or do you need active invalidation?
Inventory level changed? Invalidate immediately (event-driven). Country list changed? Wait for TTL — it's fine to be 5 minutes stale.
06
🔀
Communication & Async Design
Synchronous HTTP vs async queues vs WebSockets — when to use each
Flow design ▶
The rule: Synchronous when you need the result immediately to proceed. Async when the producer doesn't need to wait for the consumer. Real-time (WebSocket) only when you need server-push without client polling.
Communication pattern questions
✓
Are there operations where the user must wait for a result before continuing?
"Confirm payment" = synchronous, user waits. "Send email confirmation" = async, user doesn't need to wait for the email to go out. Sync = REST. Async = queue.
critical
✓
Are there fan-out operations where one event must trigger multiple downstream systems?
"Order placed" triggers payment, inventory, notification, analytics. Don't call them sequentially — one failure breaks the chain. Publish to Kafka, each system consumes independently.
✓
Does the client need server-initiated updates (real-time push)?
Chat messages, live sports scores, collaborative editing = server must push without client polling. WebSocket or SSE. 100K+ concurrent connections changes your infra significantly.
✓
Are there long-running background jobs (video encoding, report generation, data imports)?
These cannot run in a request-response cycle. They belong in a job queue (Celery, Sidekiq, BullMQ) consumed by worker processes. The API accepts the job and returns a job_id immediately.
07
🛡
Rate Limiting & Security
Protect every public surface before it's exploited
Protection layer ▶
The rule: Apply rate limiting at the API Gateway layer first. Then add endpoint-specific limits at the application layer. Never apply only at application layer — by then, the damage (DB queries, compute) is already done.
Rate limiting questions
✓
What are your sensitive endpoints? (login, OTP, payment, signup)
Login = 5 attempts/minute/IP. OTP = 3/10 minutes/phone. Payment = per-user, per-card daily limits. Each needs its own limit, its own algorithm.
critical
✓
Who is the rate limiting subject — IP, user ID, API key, or tenant?
IP-based fails at corporate NAT (500 users share one IP). User-ID is better for authenticated users. API keys for B2B. Tenant-level for multi-tenant SaaS.
✓
What happens when a user is rate-limited? Hard reject or graceful degrade?
Hard 429 for security endpoints. Cached/degraded response for read endpoints during traffic spikes. Fail-open vs fail-closed when your rate limiter (Redis) goes down.
Security questions
✓
How are users authenticated? JWT vs session tokens vs API keys?
JWT = stateless, scales horizontally, hard to revoke. Sessions = stateful, easy to revoke, needs Redis. Pick based on whether instant revocation is a requirement.
✓
Is data encrypted at rest and in transit?
HTTPS/TLS always. Sensitive fields (passwords, PII) encrypted at the application layer before storage. S3 server-side encryption for stored files.

🌍
Reliability & Observability
Design for failure — because everything fails eventually
Ops layer ▶
The rule: Every component that can fail will fail. Design every dependency with a fallback. You don't discover you need observability after the outage — you discover what you wish you had logged.
Failure modes
✓
What are your single points of failure? What fails when each dies?
Draw the dependency graph. Every node with one incoming edge is a SPOF. Add redundancy there first. The database primary is always a SPOF until you have failover configured.
critical
✓
What is your failover strategy for the database primary?
How long until a replica is promoted? Manual (minutes of downtime) or automatic (seconds)? Automatic failover needs a sentinel or cluster manager configured before the incident.
✓
Do downstream services need circuit breakers?
If Service A calls Service B and B is slow, A's thread pool fills up and A goes down too. Circuit breaker stops calls to B when it's unhealthy. Prevents cascade failure.
Observability questions
✓
What are the 3–5 metrics that would alert you of a problem before users notice?
Request error rate. p99 latency. DB connection pool utilization. Queue depth. Cache hit ratio. Each should have a threshold and an alert. Not dashboards — alerts.
✓
Can you trace a single request across all services from a single trace ID?
Distributed tracing (Jaeger, Datadog APM) propagates a trace ID through every service hop. Without it, debugging "why is checkout slow" across 8 services is guesswork.

Data Ingestion Requirements
How data enters your system — volume, velocity, variety
Entry point ▶
The 3 Vs of data: Volume (how much), Velocity (how fast), Variety (what shape). Your answers to these three questions will select your ingestion technology — Kafka vs Kinesis vs batch ETL vs direct API writes.
Volume & velocity
✓
How many events/records arrive per second at peak?
Under 1K/sec = simple. 1K–100K/sec = Kafka or managed streaming. 100K+/sec = Kafka cluster with careful partition design. This number alone determines ingestion architecture.
critical
✓
Is ingestion real-time / streaming, or batch (scheduled intervals)?
User clicks = streaming (sub-second latency needed). Monthly financial reports = batch. Many systems need both: streaming for real-time dashboards + batch for historical accuracy.
critical
✓
Can you afford to lose events, or is every event critical?
Click-stream analytics = losing 0.1% of events is fine. Financial transactions = losing one is a compliance violation. "At-most-once" vs "at-least-once" vs "exactly-once" semantics matter here.
Data shape & sources
✓
What are the data sources? (IoT sensors, app events, DB change streams, third-party APIs, file uploads)
IoT → MQTT. App events → Kafka/Kinesis. DB changes → CDC (Debezium). File uploads → S3 trigger → Lambda. Each source type has a natural ingestion path.
✓
Is the schema fixed or does it evolve? Who owns schema validation?
Evolving schemas in high-volume streaming = schema registry (Confluent Schema Registry with Avro/Protobuf). Without schema enforcement, one bad producer breaks every downstream consumer.
02
⚙
Processing Requirements
What transformations must happen to data before it's useful?
Transform layer ▶
Lambda vs Kappa architecture: Lambda runs separate batch and streaming pipelines (complex but powerful). Kappa runs everything through one streaming pipeline (simpler, fewer moving parts). Which you need depends on whether you need historical reprocessing.
Processing pattern questions
✓
Does processing need to happen within a time window of arrival? What's the latency budget?
Fraud detection: must process in under 100ms. Daily dashboard: process overnight batch is fine. Sub-second = stream processing (Flink, Spark Streaming). Minutes/hours = batch (Spark, dbt).
critical
✓
Do transformations require joining streams? Across what time windows?
"Join user click with product view within 30 minutes" = stateful stream join with 30-min window. Large windows = large state = expensive memory. Know your join windows before picking a stream processor.
✓
Do you need to reprocess historical data when logic changes?
If your ML feature logic changes and you need to recompute 2 years of history, you need a batch layer. Kafka retains data for configurable periods — reprocess by replaying from offset 0.
✓
Are there aggregations (counts, sums, averages) that must be maintained in real time?
Real-time "active users in last 5 minutes" counter = Redis HyperLogLog or Flink count aggregation. Updating a Postgres table for every event = write bottleneck at high volume. Choose the right aggregation store.
metric

Storage Architecture
Hot, warm, cold tiers — partition strategy — column vs row storage
Data layer ▶
The tiering principle: Not all data is equally hot. Recent data gets fast expensive storage. Old data gets slow cheap storage. Define the boundary between hot and cold — this drives partitioning strategy and storage cost.
Storage model questions
✓
Are queries predominantly row-based (get one record) or column-based (aggregate a column across many records)?
OLTP (get order by ID) = row-oriented storage (Postgres, MySQL). OLAP (sum revenue across all orders) = columnar storage (ClickHouse, BigQuery, Redshift). Wrong choice = 100× slower queries.
critical
✓
What is the time-to-hot-data boundary? After how long does data become cold?
Last 30 days on fast NVMe SSD. Last year on HDD. 1+ years on S3 Glacier. Cassandra lets you move old partitions to cheaper storage tiers natively. Define the boundary in SLAs.
✓
What is the primary partition key for the largest dataset?
Time-series: partition by time bucket (year/month). User data: partition by user_id range or hash. Bad partition key = one node gets all the traffic while others sit idle. Hotspot kills performance.
critical
✓
Do you need ACID on writes, or is eventual consistency acceptable?
Analytics data: eventual is fine — a 5-second lag in a dashboard doesn't matter. Financial records: ACID required — losing a transaction row is a compliance issue.
Retention & compliance
✓
How long must data be retained? Who has the right to delete it?
GDPR right-to-erasure means users can request deletion. If their user_id is a partition key, you can't just drop one row efficiently — design for soft delete + async purge, or tenant-isolated storage.
04
🔍
Query & Access Patterns
How data exits the system — who reads it, how often, how fast it must be
Read layer ▶
The rule: Design your data model around your read patterns, not your write patterns. You can always normalize data on write — but you pay for bad read models on every single query at scale.
Query pattern questions
✓
What are the top 5 queries by frequency? Write them out.
If your most frequent query is "events by user_id in the last 24h", your partition key must be user_id and your sort key must be timestamp. The query drives the schema, not the other way around.
critical
✓
Are queries ad-hoc (any filter, any column) or pre-defined (same patterns every time)?
Ad-hoc analytics = Elasticsearch or ClickHouse with wide indexes. Pre-defined queries = optimize schema specifically for those queries, avoid full-table scans. Ad-hoc at scale is expensive — cache aggressively.
✓
What is the latency budget for reads? Sub-second for user-facing, or minutes for batch reports?
User-facing dashboard under 200ms = pre-aggregate and cache, or use in-memory OLAP (Druid). Internal reports in 5 minutes = Spark batch job on data lake. Never serve raw SQL aggregation over billions of rows to users.
metric
✓
Do you need full-text search? Fuzzy matching? Faceted filtering?
These queries are not what SQL databases are built for. Elasticsearch or OpenSearch for text search + facets. Running LIKE '%keyword%' on 100M rows is a database killer.
05
📊
Replication & Partitioning Decisions
Scale reads with replicas. Scale writes and size with sharding.
Scale decisions ▶
The checklist before sharding: Index optimization → caching → read replicas → vertical scaling → archiving. Only after all five are exhausted does sharding make sense. Sharding is a permanent, expensive commitment.
Replication questions
✓
What is your data loss tolerance? (RPO — Recovery Point Objective)
RPO = how much data you can afford to lose if primary crashes. Zero = synchronous replication. Seconds = semi-sync. Minutes = async. Financial: zero. Analytics: minutes is fine. Pick your replication mode from RPO.
critical
✓
What is your maximum acceptable downtime if primary fails? (RTO — Recovery Time Objective)
RTO = time from failure to recovery. 30 seconds = automated failover pre-configured. 30 minutes = manual promotion. Your RTO determines whether you invest in Patroni/Orchestrator or handle failover manually.
critical
✓
Are reads predominantly on recent data or historical? Can reads tolerate replication lag?
"Show me last week's analytics" from a replica = fine, 1-second lag doesn't matter. "Show me the balance right after my transfer" = must go to primary. Route reads intelligently based on staleness tolerance.
Sharding questions
✓
Is write throughput saturating a single primary? Have you measured this?
Sharding adds irreversible complexity. "It might get slow" is not enough. Measure CPU, disk I/O, and write latency on the primary first. Add the optimization checklist before sharding.
✓
Will your most common queries ever need to span multiple shards?
Cross-shard queries = scatter-gather = expensive. If "top 10 customers by revenue globally" is a common query, sharding by customer_id makes it require hitting every shard. Design to avoid this.
critical
✓
Is your shard key at risk of creating hotspots?
Timestamp-based shard key = all writes always hit the latest shard. user_id hash = uniform distribution but no range queries. Celebrity problem: one shard key (a viral user) gets 10× the traffic. Plan mitigation.

🔄
Pipeline Reliability & Data Quality
What happens when something breaks in the middle of the pipeline
Reliability ▶
The rule: Pipelines fail silently. A queue that's processing 10% of events looks healthy until you notice metrics are off by 90%. Design explicit failure detection, dead letter queues, and idempotent consumers from day one.
Failure handling questions
✓
What happens to an event if a consumer fails mid-processing?
If consumer crashes after reading but before writing → event lost (at-most-once). Must re-read from queue → event reprocessed (at-least-once). Consumer must be idempotent: processing the same event twice produces the same result.
critical
✓
Do you have a Dead Letter Queue (DLQ) for events that fail after max retries?
Without DLQ: one poison-pill message blocks the entire queue indefinitely. With DLQ: bad events are parked, main queue continues. DLQ must be monitored with alerts — a growing DLQ means systematic failure.
✓
How do you detect and measure data quality degradation?
Row count checks between source and destination. Null rate monitoring on critical columns. Schema drift alerts. A data quality check that runs after every pipeline step is worth more than 10 dashboards.
✓
Can the pipeline be paused and resumed without data loss? Can you replay from a checkpoint?
Kafka offsets give you this for free — consumers commit their position; restart from last committed offset. Batch pipelines need checkpointing (last successfully processed file/date). Know your replay strategy before you need it.
The meta-principle behind both blueprints: In system design, every tool is an answer to a question. Kafka answers "how do I decouple these services?" Redis answers "how do I serve this in under 1ms?" Cassandra answers "how do I write 100K records/sec without sharding complexity?" Before picking any technology, make sure you have already asked — and answered — the question it solves. A technology chosen without a backing question is architecture astronautics.
