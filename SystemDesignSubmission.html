Task 1 — URL Shortener Lite

Overview

A lightweight URL shortening service. Before writing any code, the most important design question was: which operation happens more often — shortening or redirecting? Redirects vastly outnumber shortens, so the GET path must be as fast as possible.

Step 1 — Base62 Encoding

Base62 uses digits (0-9), uppercase (A-Z), and lowercase (a-z) — 62 characters total. With 6 characters: 62⁶ ≈ 56 billion unique IDs. The URL is hashed with SHA-256 and the first 48 bits of the hash are encoded as Base62, giving a deterministic, collision-resistant short ID.

Code — Base62 encoder

BASE62 = "0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz"

def to_base62(number, length=6):
    """Convert an integer to a Base62 string of fixed length."""
    result = ""
    while len(result) < length:
        result = BASE62[number % 62] + result
        number //= 62
    return result[:length]

def generate_short_id(url, attempt=0):
    payload  = url if attempt == 0 else f"{url}:{attempt}"
    hex_hash = hashlib.sha256(payload.encode()).hexdigest()
    number   = int(hex_hash[:12], 16)   # first 48 bits
    return to_base62(number)
Why hash the URL instead of generating a random ID? The same URL always produces the same hash → automatic duplicate detection with just one indexed lookup. No extra "has this URL been seen?" query needed on the happy path.

Step 3 — API Endpoints

Code — POST /shorten

@app.post("/shorten")
def shorten():
    url      = request.get_json().get("url", "").strip()
    url_hash = hashlib.sha256(url.encode()).hexdigest()

    # Duplicate check — same URL always gets same short link
    existing = db.execute("SELECT short_id FROM urls WHERE url_hash = ?",
                          (url_hash,)).fetchone()
    if existing:
        return jsonify({"short_id": existing["short_id"], ...})

    # Generate short ID; retry loop handles hash collisions
    attempt = 0
    while True:
        short_id = generate_short_id(url, attempt)
        if not db.execute("SELECT 1 FROM urls WHERE short_id = ?",
                          (short_id,)).fetchone():
            break
        attempt += 1

    db.execute("INSERT INTO urls (short_id, original_url, url_hash) VALUES (?,?,?)",
               (short_id, url, url_hash))
    db.commit()
    return jsonify({"short_id": short_id, "short_url": f"{BASE_URL}/{short_id}"}), 201

Code — GET /<short_id>

@app.get("/<short_id>")
def redirect_url(short_id):
    row = db.execute("SELECT original_url FROM urls WHERE short_id = ?",
                     (short_id,)).fetchone()
    if not row:
        return jsonify({"error": "Short URL not found"}), 404
    return redirect(row["original_url"], code=302)

Task 2 — Rate Limiter Mini

Flask · In-memory dictionary · Sliding Window · 5 requests / 60 seconds / IP

Overview

A rate limiter applied to GET /data. Each user (identified by IP) is allowed 5 requests per 60-second sliding window. State is stored in an in-memory Python dictionary that maps each IP to a list of timestamps.

Architecture Diagram

GET /data │ ▼ ┌──────────────────────────────────────────────────┐ │ is_rate_limited(ip) │ │ │ │ in-memory dict: │ │ { "1.2.3.4": [t1, t2, t3, t4, t5], ... } │ │ │ │ Step 1 ── Prune timestamps older than 60s │ │ timestamps = [t for t in timestamps │ │ if t >= now - 60] │ │ │ │ Step 2 ── Count remaining (= requests in window)│ │ count = len(timestamps) │ │ │ │ Step 3 ── Decision │ │ count >= 5 ? │ │ YES → 429 Too Many Requests │ │ NO → append now, allow request │ └──────────────────────────────────────────────────┘ │ │ ▼ ▼ 200 OK 429 Too Many Requests "Here is your data!" "Try again later."

Step 1 — Fixed Window vs Sliding Window

The requirement says "5 requests per minute," but a naive Fixed Window resets at :00 every minute. A user can fire 5 at :59 and 5 more at :01 — 10 requests in 2 seconds, neither window reports over the limit.

Fixed Window (broken at boundary): Minute 1: [─────────── 5 requests at :58, :59 ──] ← allowed Minute 2: [5 requests at :00, :01 ───────────────] ← allowed ↑ 10 requests in 3 seconds — not rate limited! Sliding Window (correct): At any moment, look back exactly 60 seconds from NOW. Always at most 5 requests in any 60-second span. ✓

Step 2 — In-Memory Dictionary Implementation

Code — is_rate_limited()

# In-memory store — maps IP → list of Unix timestamps
request_log: dict[str, list[float]] = {}

LIMIT      = 5   # requests allowed
WINDOW_SEC = 60  # window size

def is_rate_limited(ip: str) -> tuple[bool, int]:
    now          = time.time()
    window_start = now - WINDOW_SEC

    timestamps = request_log.get(ip, [])

    # Step 1 — prune expired timestamps (the "sliding" in sliding window)
    timestamps = [t for t in timestamps if t >= window_start]

    count = len(timestamps)

    if count >= LIMIT:
        # Return seconds until oldest slot frees up
        retry_after = int((timestamps[0] + WINDOW_SEC) - now) + 1
        request_log[ip] = timestamps   # save pruned list
        return True, retry_after

    # Allow: record this request
    timestamps.append(now)
    request_log[ip] = timestamps
    return False, LIMIT - len(timestamps)

Code — /data route

@app.get("/data")
def data():
    ip = request.remote_addr or "unknown"
    limited, value = is_rate_limited(ip)

    if limited:
        return jsonify({
            "error": "Too Many Requests",
            "message": "Try again later.",
            "retry_after_seconds": value
        }), 429

    return jsonify({
        "message": "Here is your data!",
        "requests_remaining": value
    })

Why the retry_after_seconds is not always 60 When the 6th request arrives, the oldest timestamp might be from 10 seconds ago — meaning that slot will expire in 50 seconds, not 60. retry_after = timestamps[0] + 60 - now gives the exact time until 1 slot frees up, anchored to the first request in the window.

Step 2 — Round Robin Algorithm

Code — load_balancer.py

SERVERS = ["http://localhost:5010", "http://localhost:5011", "http://localhost:5012"]
request_counter = 0

def get_next_server() -> str:
    global request_counter
    server = SERVERS[request_counter % len(SERVERS)]  # modulo = cyclic rotation
    request_counter += 1
    return server

@app.route("/", defaults={"path": ""}, methods=["GET","POST","PUT","DELETE"])
@app.route("/<path:path>",             methods=["GET","POST","PUT","DELETE"])
def proxy(path):
    target = get_next_server()
    print(f"[LB] Request #{request_counter} → {target}")

    resp = requests.request(         # forward via Python requests library
        method  = request.method,
        url     = f"{target}/{path}",
        headers = {k: v for k, v in request.headers if k != "Host"},
        data    = request.get_data(),
        timeout = 5,
    )
    return Response(resp.content, status=resp.status_code,
                    headers={"X-Served-By": target})
Why modulo arithmetic? servers[counter % n] is an elegant, zero-state solution for cyclic rotation. A single incrementing integer drives infinite round-robin distribution across any number of servers. No queue, no circular buffer needed.

Production Approach vs Assignment Approach — Trade-offs

This section compares the simplified approach used in this assignment with a production-ready implementation, explaining when each is appropriate.

Task 1 — URL Shortener

DimensionAssignment (SQLite)Production (PostgreSQL + Redis)
StorageSQLite — single file, no server needed, perfect for learning and low trafficPostgreSQL — handles concurrent writes, row-level locking, replication
Read speedDirect DB lookup on every redirect — adequate for hundreds of req/sRedis cache in front of DB — microsecond lookups, handles millions of req/s
ScalabilitySingle-process only — can't scale horizontallyCache layer means any number of app servers share the same fast read path

Task 2 — Rate Limiter

DimensionAssignment (in-memory dict)Production (Redis sorted set)
State locationPython process memory — lost on restartRedis — persists across restarts, shared across all server instances
Multi-serverEach server has its own counter — a user can bypass the limit by hitting different serversAll servers share one Redis instance — true global rate limiting
AtomicitySingle-process Python GIL prevents race conditions in development, but not safe under true concurrencyRedis MULTI/EXEC pipeline is atomic — no race conditions under any load
MemoryDict grows unbounded if IPs never expire — needs manual cleanupRedis TTL auto-expires old keys — zero maintenance
When to useUnderstanding the algorithm, single-server dev environmentsAny multi-instance deployment or production API
Learning progression The in-memory dictionary makes the algorithm transparent — every step is visible in plain Python. Redis abstracts away the storage but adds reliability. The right approach is to understand the dict version first (as done here), then graduate to Redis for production.

Final Notes

Key Learnings

  1. Read/write ratio drives architecture. The URL shortener is ~8:2 reads to writes. This one insight justifies every optimisation — caching, read-optimised indexes, and the Cache-Aside pattern.
  2. Sliding Window eliminates the Fixed Window boundary exploit. Resetting at fixed intervals lets users double their effective rate at the minute boundary. A sliding window anchored to "now - 60s" eliminates this completely.
  3. Hash-based short IDs give free duplicate detection. The same URL always hashes to the same ID — no extra lookup needed to check for existing entries.
  4. Modulo arithmetic is all Round Robin needs. servers[counter % n] distributes traffic perfectly with a single integer and zero additional data structures.
  5. In-memory state has invisible multi-server failure modes. The dict rate limiter works correctly on one server but silently fails when load-balanced — a critical lesson in why distributed systems need shared state.
  6. Retry-After should be anchored to the oldest request, not the current time. If the first of 5 requests was 50 seconds ago, the real retry is 10 seconds — not 60. Computing timestamps[0] + 60 - now gives the correct answer.
  7. Always verify technology constraints before starting implementation. Strong architectural knowledge is only valuable when applied within the given requirements. Production thinking is a tool, not a default.

Challenges Faced and How They Were Solved

ChallengeWhat went wrongHow it was solved
Initial submission used wrong tech stack Built with Node.js, PostgreSQL, and Redis — production-grade but outside assignment constraints Rewrote all three tasks in Python using Flask, SQLite, and in-memory dict as specified. Kept architectural thinking, changed the implementation layer.
Retry-After countdown started from 60s even mid-window The UI hardcoded retryAfter = 60 as a fallback when the server didn't return the value. Also, the 5th successful request returned no retryAfter at all. Backend now computes timestamps[0] + 60 - now (oldest entry in window) and returns the precise remaining seconds. Frontend reads from the response instead of hardcoding.
Duplicate URL detection for very long URLs Indexing raw URL strings up to 2000+ characters is slow and can exceed SQLite limits. Store and index the SHA-256 hash (fixed 64 chars) of the URL instead of the URL itself. The hash is uniqueness-preserving and always the same length.
Hash collision on short ID generation Two different URLs could produce the same 6-character Base62 ID. Added a retry loop: on collision, append an attempt counter to the URL before hashing (f"{url}:{attempt}"), shifting the hash output until a unique ID is found.
Rate limiter in-memory dict grows without bound IPs that haven't made a request in hours still occupy memory. Timestamps are pruned on every request (entries older than 60s are removed). If the resulting list is empty, the key can also be deleted. In production, Redis TTL handles this automatically.
